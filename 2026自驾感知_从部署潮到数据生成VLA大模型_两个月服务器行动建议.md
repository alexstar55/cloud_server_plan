# 2026 自驾感知：从“部署潮”到数据生成 / VLA / 大模型——判断校准与两个月行动建议

> 目标：让工作技能的积累能持续“供养”你的数理兴趣：既能产出对业务有用的工程资产（数据/评测/训练管线），又能在方法层面保持数学含量（几何、优化、概率、表示与对称性的直觉在模型里落地）。

---

## 1) 你的行业观察对吗？（我给出的校准版本）

### 1.1 “岗位偏部署”总体是对的，但不是“数学退潮”

**对的部分：**

- 自驾感知的许多团队进入“产品化/规模化交付”阶段后，岗位确实更偏：
  - 端侧/车规部署（TensorRT/ONNX、量化、剪枝、算子融合、调度、CUDA kernel、内存/带宽优化）
  - 工程化闭环（数据、训练、回归、监控、A/B、灰度）
  - 成本与稳定性（吞吐、延迟、可解释性、安全合规）

**需要补充的部分：**

- “部署”本身不等于低数学含量：很多核心问题变成了**约束优化**（延迟/能耗/精度/鲁棒）与**统计学习**（漂移、置信度、校准、长尾），数学味道依然浓。

### 1.2 “数学相关度高集中在数据生成 / VLA / 大模型”大体合理，但可以再扩一圈

你点到的方向确实更靠近 2026 的“上升叙事”：

- **数据生成/合成数据/仿真到现实**：分布匹配、域随机化、反事实、可识别性、因果/统计。
- **VLA/具身智能**：多模态表征、序列决策、世界模型、泛化。
- **大模型（多模态基础模型、视频/3D/BEV world model）**：表示学习、对称性、泛化理论。

但别漏掉几类“数学含量同样高、且更贴业务”的方向（往往更容易在公司内形成影响力）：

- **评测与统计可靠性**：置信区间、显著性、漂移检测、风险度量、校准（calibration）、OOD。
- **不确定性与安全**：贝叶斯近似、conformal prediction、风险敏感训练、分布鲁棒优化（DRO）。
- **优化与系统**：大规模训练的数值稳定性、混合精度、并行策略、吞吐/延迟优化。
- **几何与对称性**：SE(2)/SE(3) 等变（equivariance）、几何先验、图与场（field）表征。

结论：你的观察方向感很对；更精确的说法是——**“研发价值开始向数据与评测闭环 + 基础模型能力 + 可规模化工程系统”集中**，数学会更多以“统计/优化/几何约束”的形式出现。

## 2) 你的计划合理吗？（从资源与时间约束倒推）

你有：

- 一台带 GUI 的云服务器
- 双 5880 级别 GPU（大概率显存较大，但仍然不是训练超大模型的集群）
- 两个月使用期、流量不限

### 2.1 合理的部分

- 两个月窗口很适合做：
  - 复现一个“代表性方法”并做工程化沉淀（数据/评测/训练脚本）
  - 在合成数据或仿真上跑通闭环（从生成到训练再到评测）
  - 做 1–2 个高质量的 ablation + failure case 分析

### 2.2 需要警惕的部分：清单太长、目标容易漂

你列的 SparseDrive、VGGT/DriveVGGT、Sparseworld-TC、alpaSim/Alpamayo、ApaVLA 等，覆盖面很广。两个月如果同时追多个方向，风险是：

- 每个 repo 都“跑起来了”，但没有形成可复用资产
- 最终没有一条能对业务产生稳定影响（评测、数据、收益点说不清）

**建议原则：两个月只做“1 条主线 + 1 条副线”。**

- 主线：能形成公司可复用资产（通常是数据或评测闭环）
- 副线：跟基础模型/具身方向对齐（为你的长期兴趣与职业弹性加码）

## 3) 两个月如何选题：把“数学感”和“职业回报”同时最大化

### 3.1 我建议你把主线定为：数据生成与评测闭环（可迁移、可复用）

原因：

- 对任何模型范式都通用（稠密/稀疏、2D/3D/BEV、VLA）
- 最容易在公司内部拿到认可（能提升回归效率与长尾覆盖）
- 数学含量高（分布、统计、优化、因果、信息论直觉）

落地形态最好是：

- 一个“可配置”的数据生成/采样器（长尾驱动）
- 一套评测协议（场景分类、关键指标、置信度/校准、统计显著性）
- 一条可复现训练流水线（baseline + 你做的增量）

### 3.2 副线：VLA/世界模型方向选一个“可跑通的最小闭环”

两个月不建议从零训练超大 VLA，而是：

- 选一个开源路线做“对齐 + 评测 + 小规模微调”，目标是产出：
  - 可复用的多模态数据格式/对齐工具
  - 可衡量的指标（任务成功率、泛化、鲁棒性、长尾）

## 4) 两个月执行计划（按周里程碑，可交付）

### Week 0（开服前几天）：设计实验与产物形态（最关键）

**交付物：**

- 一个 1 页的实验设计文档：
  - 目标：提升什么（长尾召回/误检/定位/跟踪稳定性/闭环效率）

  - 预算：两个月 GPU 小时如何分配
- 选定主线 1 个 repo + 副线 1 个 repo（其余进入“候选池”）

### Week 1：环境与复现（只追求可重复）

**交付物：**

- 统一的运行入口：`make train` / `make eval` 或 `bash run_*.sh`
- 实验记录：W&B 或 MLflow（至少把 config、git hash、metrics 固化）
- 复现 baseline：能在小数据上跑通 train + eval

### Week 2：评测框架与数据 schema 固化

**交付物：**

- 数据 schema：传感器、标注、合成参数、场景标签、版本号
- 评测框架：
  - 指标计算（主指标 + 诊断指标）
  - 场景切片（白天/夜晚/雨雾、拥挤度、遮挡、速度分桶等）
  - 简单的统计检验（bootstrap 置信区间或配对检验）

### Week 3–4：数据生成/采样策略迭代（主线发力）

**交付物：**

- 合成数据 pipeline V1：
  - 可参数化生成
  - 与真实数据格式对齐
  - 能自动产出场景标签与元数据
- 采样策略：
  - 长尾优先（hard example mining / uncertainty-guided sampling）
  - 覆盖率度量（场景分布对齐）

### Week 5–6：把数据增量转成模型收益（闭环）

**交付物：**

- 至少 2 组清晰 ablation：
  - 只改数据
  - 数据 + 训练策略（例如 loss reweight、curriculum）
- 失败案例库：可视化页面/Notebook（GUI 服务器优势）

### Week 7：副线（VLA/多模态）最小闭环

**交付物：**

- 一个可复用的多模态对齐/打包工具（视频/BEV/轨迹/语言指令）
- 1 个小规模微调或 adapter/LoRA 实验 + 可解释的评测

### Week 8：收敛与“对外可讲”的结果

**交付物：**

- 一份内部分享材料：
  - 你做了什么、为什么有效
  - 对业务的价值（效率/覆盖/指标）
  - 下一步路线（如果继续租服务器，最值得加码的是什么）
- 一个可交接的 repo：README、环境、runbook、数据版本说明

## 5) 关于你提到的方向：如何筛选与定位（避免两个月被 repo 吞掉）

我不对这些项目逐个“拍板好坏”（不同公司数据/任务差异极大），但给一套筛选标准，能快速判断是否值得投入：

### 5.1 选择标准（建议按优先级）

1. **可评测性**：有没有清晰的 eval protocol？你能否在 1 周内搭起离线评测？
2. **数据契合**：需要的数据类型与你公司可获得的数据匹配吗（多传感器/标注粒度/时序长度）？
3. **可控计算量**：能否在双卡上做有意义的实验（微调、蒸馏、adapter、部分模块训练）？
4. **可解释增量**：你能做 ablation，说清“为什么变好/哪里变坏”吗？
5. **可复用资产**：即使模型不 SOTA，数据管线/评测/可视化是否能沉淀？

### 5.2 对你列的方向的“定位建议”

- 类 SparseDrive / 稀疏建模路线：很适合落在“高效表示 + 计算受限”的现实问题上；把它当作主线时，务必先把评测与数据 schema 固化。
- VGGT/DriveVGGT 一类：更偏表示学习与任务统一；适合作为副线做“小规模对齐与评测”。
- Sparseworld-TC、alpaSim/Alpamayo：如果它们能显著提升合成数据质量或场景覆盖，它们更适合放在主线的数据生成段。
- ApaVLA：适合作为副线，但要把目标限定为“数据格式/对齐工具 + 小微调 + 明确评测”。

## 9) 战略调整：面向未来 3–5 年的“可迁移技术资产”

你这次澄清非常关键：**你要积累的是“下一份工作也通用”的能力与作品**，而不是把某个公司当下栈（例如 BEVFusion）做到极致。

因此下面把策略从“公司内闭环”调整为“长期可迁移资产”。你仍然可以用 BEVFusion 作为一个 baseline/对照组，但它不再是主叙事。

### 9.1 2026–2030：感知/具身赛道里最可迁移的 5 类硬资产

这些资产在任何平台都值钱，并且天然带数学含量（统计/优化/几何/表示）：

1. **评测与可靠性（Evaluation & Reliability）**
  - slice-based eval、统计显著性、漂移检测、校准（ECE/NLL）、OOD/长尾
  - 这是“把模型研究变成工程收益”的核心杠杆，也是很多团队缺口
2. **数据引擎（Data Engine）**
  - hard example mining、active learning、合成/仿真数据的分布控制与版本管理
  - 数据资产与方法无关，能跨模型范式复用
3. **几何化表征（Geometry-first Representation）**
  - BEV 是入口，但未来更核心的是：occupancy/flow/scene dynamics、map prior、多传感器一致性
4. **基础模型适配能力（Foundation Model Adaptation）**
  - 多模态（视频+3D+语言）对齐、LoRA/adapter、蒸馏、评测协议
  - 不是“训练巨模型”，而是“把模型变成可用系统”
5. **效率与部署（Efficiency）**
  - profiling、吞吐/延迟、TensorRT、量化/蒸馏、稀疏/缓存策略
  - 很多公司把它当“部署”，但做深了是硬核系统+优化

你要做的，是在两个月里产出至少一件“能作为作品集”的资产：**可复现、可评测、可解释、可扩展**。

### 9.2 两个月云服务器：最推荐的“作品集式主线”（1 个就够）

在每周 3–8 小时的约束下，你不应该追多个模型 repo；你应该做一个“平台无关”的工程/研究中间层：

**主线建议（强烈推荐）：Perception Eval + Data Engine（面向 BEV/3D/时序的通用框架）**

它的定位是：

- 输入：NuScenes 格式（你已有 8k 帧，且可直接叠加公开 nuScenes 数据）
- 输出：
  - 一套可复用的评测与切片（误检、远距、稳定性/时序一致性、校准）
  - 一个 hard negative miner + 采样策略（能驱动下一轮训练/数据收集）
  - 一个可视化报告（GUI 优势）
- 模型侧：不绑定某个模型；支持“插件式接入”
  - 你可以接入 BEVFusion、SparseDrive、occupancy baseline 作为对照

这条主线对你跳槽非常友好：

- 面试能讲清“我如何定义问题、构建评测、用数据闭环拿到收益”
- 不依赖公司私有数据（你可以用公开数据演示框架，用公司数据验证私下收益）

### 9.3 副线（可选）：为 L4 方法做“占用/运动”评测协议 + 最小实验

如果你希望在方法趋势上更靠前，副线不追“新模型”，追“新输出形态”的可评测性：

- 产出一套 occupancy/flow 的评测与可视化（IoU、EPE、时序一致性、规划代理指标）
- 接一个开源 baseline 跑通并做 1 次小规模微调/adapter

这样你既把自己放进 L4 方法语境，又不会被训练成本拖垮。

### 9.4 8 周里程碑（为“作品集交付”重新设计）

目标不是“跑通训练”，而是“产出可复用资产 + 可讲的证据链”。

**Week 0（开服前）**

- 定义作品集产物：一个 repo + 一份技术报告（README+图表）
- 明确评测口径（至少 4 类）：
  - 误检（FP taxonomy）
  - 远距（distance buckets）
  - 稳定性（temporal consistency：抖动/断裂/一致性）
  - 校准（ECE/NLL，最好再加可靠性图）

**Week 1**

- 完成 NuScenes 数据适配与基础评测脚手架（能跑通一份 baseline 结果的导入）
- 把实验可复现做成默认（config、seed、版本号、结果落盘）

**Week 2**

**Week 3**

**Week 4**

- 定义并实现“稳定性指标”并在一个 baseline 上跑出结果
  - 例如：box center/velocity 的帧间方差、track 断裂率（若有 tracking）

**Week 5–6**

- 做一次真正的闭环：
  - 用 miner 产出的样本策略做一次数据重采样/增量训练（或最小微调）
  - 给出统计上可信的对比（bootstrap CI 或配对检验）

**Week 7**

- 副线可选：接入 occupancy/flow baseline（只需跑通 eval+可视化 + 1 次小实验）

**Week 8**

- 完成“可对外讲”的报告：
  - 你解决了什么评测/数据问题
  - 哪些切片显著改善、哪些变差
  - 下一步如何把它扩展到更先进模型（SparseDrive / 世界模型 / VLA）

### 9.5 开服前这几天：除了环境脚本，最值得做的准备（面向作品集）

按回报排序：

1. 画出你的“评测 schema”：输入什么（预测/GT/元数据），输出什么（指标/切片/报告）。
2. 把切片字典先写出来（哪怕手工）：远距、遮挡、反光、雨夜、静态物体、拥挤度。
3. 把你们 8k NuScenes 数据做一次**分布体检**（每类样本数、距离分布、场景覆盖）。
4. 选定并下载公开数据/权重（nuScenes + 1–2 个 baseline 权重），为开服后节约时间。
5. 决定报告形态：
  - 一键生成的 HTML 报告（更像工程资产）
  - 或 notebook（更像研究资产）
6. 预设“止损门槛”：
  - 任何模型侧复现超过 2 天仍跑不通就换 baseline，不让 repo 吃掉两个月
7. 设计你的对外叙事：

### 9.8 Notebook 大纲（直接照着写，保证“论文味”）

一份面试友好的 research notebook，建议固定结构：

1. **Problem Statement**：定义你解决的问题（误检/远距/稳定性/校准）与现实约束（算力、数据、时序）。
2. **Dataset & Protocol**：
   - 数据版本号、样本量、距离分布、场景覆盖
   - 评测口径（你如何定义 TP/FP/FN；哪些类/哪些距离算“远距”）
3. **Metrics**：
   - 主指标（例如 mAP/NDS 或你自定义的风险指标）
   - 诊断指标：FP taxonomy、距离分桶、稳定性、校准
4. **Slice-based Evaluation**：
   - 先给全量指标，再给关键切片
   - 每张图都回答一个问题（例如：远距 60m+ 的 FP 来源是什么？）
5. **Failure Analysis**：
   - Top-K failure 可视化
   - 聚类/归因（反光/遮挡/雨夜/静态结构等）
6. **Data Engine Loop**：
   - hard negative miner 的定义
   - 闭环策略（重采样/重权重/增量数据）
7. **Ablation & Statistics**：
   - 只改一个变量
   - bootstrap 置信区间、配对检验（让结论“站得住”）
8. **Calibration & Reliability**：
   - reliability diagram、ECE/NLL
   - 按切片的校准差异（这点很能体现你对“可靠性”的理解）
9. **Takeaways & Next Steps**：
   - 你学到了什么（可迁移）
   - 下一步如何接更先进模型（稀疏/occupancy/世界模型/VLA）

### 10.1 Baseline A（快速稳定）：BEVFormer（Occ3D-nuScenes 版本）

定位：

- 用于快速产出第一批预测 → 让你的切片/统计/作图框架尽快跑起来。

为什么它适合做作品集 baseline：

### 10.2 Baseline B（更前沿）：CVT-Occ（Occ3D-nuScenes 版本）

定位：

- 用于证明你的 eval/data-engine 框架能迁移到更强方法，并能做 ablation/显著性。

### 10.3 数据：Occ3D-nuScenes 的关键事实（你写进 protocol 与 notebook）

Occ3D 官方给出的 nuScenes 体素配置（用于你在 notebook 里写清楚协议）：

- voxel size：$[0.4m, 0.4m, 0.4m]$
- range：$[-40m, -40m, -1m, 40m, 40m, 5.4m]$
- volume：$[200, 200, 16]$
- classes：0–17，其中 17 表示 `free`
- 每帧 `labels.npz` 里至少包含：
  - `semantics`：体素语义标签
  - `mask_lidar`：LiDAR 可观测 mask
  - `mask_camera`：相机视角可观测 mask（**评测用它过滤不可见体素**）

作品集的“严谨性”关键点：

- 任何 mIoU/分桶统计都必须明确：是否只在 `mask_camera==1` 的体素上计算。

### 10.4 统一的预测导出 schema（Occupancy / Flow 都能装下）

你需要一个模型无关中间格式，把不同 repo 的输出统一起来。建议每帧一个 `npz`，外加一个 `meta.json`。

**每帧 `pred_occ.npz`（建议字段）：**

- `logits`：`float32`，形状 `[Z, Y, X, C]`（或 `[C, Z, Y, X]`，但你要在 meta 里写清）
- `probs`：可选（若你想省空间就只存 logits）
- `semantics_pred`：`uint8`，形状 `[Z, Y, X]`（`argmax` 结果，便于快速可视化）
- `flow`：可选，`float16/float32`，形状 `[Z, Y, X, 2]`（BEV 平面速度/位移；没有就不写）

**配套 `meta.json`（每个预测目录一份）：**

- `dataset`：`"Occ3D-nuScenes"` 或你自定义
- `voxel_size`：`[0.4,0.4,0.4]`
- `pc_range`：`[-40,-40,-1,40,40,5.4]`
- `volume`：`[200,200,16]`
- `class_names`：长度 `C`
- `axis_order`：例如 `"ZYXC"`
- `free_class_id`：例如 `17`

**重要约定（写进 protocol）：**

- 统一坐标：ego 坐标系下 $x$ 前、$y$ 左、$z$ 上；体素索引到空间坐标的映射必须在 meta 里固定。
- 若做 flow：flow 的方向与单位（m/s 或 m/step）必须写明。

### 10.5 评测口径（mIoU + slice-wise + 可靠性）

最小可交付评测：

1. **mIoU（masked）**：只在 `mask_camera==1` 的体素上算 mIoU（Occ3D 官方口径）。
2. **slice-wise mIoU**：至少按高度（地面附近 vs 高处）与距离（近/中/远）做切片。
3. **free vs occupied**：单独报告 `free` 的 IoU，以及 `occupied` 的平均（很多方法会“偏向 free”）。
4. **校准**：
   - occupancy 的校准可以先做“top-1 probability 的 reliability diagram”（按体素采样）
   - 或把它简化成二分类：occupied vs free 的 ECE

关于 flow：

- 两个月内把 flow 做成“可复现论文级别”很难（因为需要明确 GT 或可对齐的 proxy）。
- 更务实的方式是：
  - 先把 flow 放进 schema，但第 1 版只做**时序一致性指标**（例如 consecutive frames 的 occupancy 变动率、temporal IoU）
  - 等你有合适 GT/伪 GT 再补 EPE 等指标

你有外网与不限流量，这里给你一个“1 周出图”的最短路径：
